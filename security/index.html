Reality check

No internet messaging system can guarantee universal untraceability or withstand every endpoint compromise. ReDoor’s direction is to reduce attack surface, raise attacker cost, and fail closed at unsafe boundaries.

Security Goals

What the project is trying to protect.

Goals

  • Protect message confidentiality from infrastructure operators.
  • Reduce social-graph leakage at the relay boundary.
  • Keep client-side sensitive state in volatile memory.
  • Detect tampering or reordering through hash evidence.
  • Enforce strict transport hardening with TLS, pinning, and auth controls.

Current protection focus

  • Relay-host compromise should not reveal plaintext content.
  • Network replay and malformed-input paths should hit explicit validation gates.
  • Background, resign, terminate, and duress paths should clear sensitive state.
  • Security-relevant changes should leave behind tests, docs, telemetry, and rollback notes.

Threat and Claim Boundaries

What ReDoor does not claim to solve.

  • Global passive traffic analysis is fully solved today.
  • Compromised kernels, basebands, or hardware roots can be defeated by app logic alone.
  • Hash-only blockchain evidence is metadata-free.
  • ReDoor is already operationally as mature as the largest deployed messengers.

Status Taxonomy

Current controls versus ongoing or future work.

Implemented

Shipping baseline

  • Transport and relay hardening TLS relay posture, optional HMAC auth, anti-replay checks, fetch-once normal blobs, and abuse controls.
  • Volatile state model RAM-only policy direction in the iOS app plus runtime wipe and duress handlers.
  • Security process gates Standards profile, release sign-off workflow, security-relevant PR checklist, and reproducibility checks.
  • Metadata-hardening controls Cover traffic, strict anonymity mode, bounded schedule jitter, multi-relay retrieval, and anti-correlation scoring.
  • Zero-click boundary work Untrusted parser isolation boundary, policy allowlist, fail-closed worker lifecycle, and parser-inventory gate.
  • PQ recovery baseline Forced rekey and protocol-version binding are documented as merged production baseline work.
In Progress

Active public work

  • Standards-aligned auth and validation evidence ASVS V2 and V5 negative-path coverage are still marked partial in the control matrix.
  • Mobile resilience evidence MASVS runtime compromise evidence is active work rather than closed status.
  • Managed key custody Provider-backed hooks exist, but managed or hardware-backed signing and audit coverage remain partial.
  • PIR or proxy deployability PIR or proxy retrieval remains an opt-in research track gated by tasks
Planned

Next hardening direction

  • Secure enclave backed identity mode Hardware-backed long-lived key storage is listed as a next-step endpoint hardening direction.
  • Optional runtime attestation Service binary or config attestation remains a future operational control.
  • External formal protocol review Formal verification depth and broader external assurance remain part of the longer-term program.

Hardening Tracks

Beyond message secrecy alone.

Workstream

Endpoint hardening

Zero-click isolation, parser minimization, lockdown profiles, and memory-hygiene audits.

Workstream

Metadata privacy

Route diversity, schedule jitter, mirrored retrieval, and PIR or proxy research.

Workstream

Protocol evolution

Explicit PQ policy negotiation, forced rekey, and future triple-ratchet-style recovery work.

Workstream

Operational integrity

Reproducible builds, provenance, drill programs, release sign-off, and external audit packaging.
