Core Differentiators

A technical product site for a protocol-shaped system.

ReDoor is not presented as a generic chat app. The public-facing site centers on its actual trust posture: untrusted transport, volatile client behavior, integrity evidence, and explicit hardening work.

Why it matters

Relay-blind forwarding

ReDoor’s relay is transport-only. It stores opaque blobs in memory, validates transport controls, and is intentionally not trusted with plaintext content.

Read more

Why it matters

Volatile client state

Sensitive runtime state stays in memory and is explicitly wiped on background, resign, terminate, and duress flows in the current design.

Read more

Why it matters

Metadata-resistance direction

Strict anonymity mode, cover traffic, bounded schedule jitter, rotating mailbox handles, and multi-relay retrieval are treated as an ongoing engineering track rather than a slogan.

Why it matters

Tamper evidence without plaintext

Hash commitments go to the blockchain evidence plane while plaintext and ciphertext stay off-chain.

Architecture At A Glance

Architecture at a glance

The product splits into five runtime planes: client, iOS shell, relay transport, directory discovery, and evidence submission. Each plane has a narrow responsibility and a specific trust boundary.

Plaintext should exist only inside the device boundary.

Relay transport is untrusted and content-blind.

Directory responses must be signed and verified.

The blockchain plane records commitments for integrity, not secrecy.

client/

Client runtime

Rust cryptographic engine, transport clients, diagnostics, FFI.

RedoorApp/

iOS app

SwiftUI shell with lifecycle wipes and secure-network validation.

relay-node/

Relay node

TLS relay with HMAC, replay defense, in-memory queues, abuse controls.

directory-dht/

Directory DHT

Signed key resolution, TTL prekeys, sequence and lease ownership.

blockchain-node/

Blockchain node

Commitment ledger for tamper evidence without plaintext storage.
