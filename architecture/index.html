Architecture

Architecture

ReDoor splits into five runtime planes so trust boundaries stay explicit: client runtime, iOS shell, relay transport, directory discovery, and blockchain-backed evidence.

Source alignment

Derived from README, SYSTEM_DESIGN, and docs/architecture

This page stays grounded in the public repository’s current component responsibilities and message flow.

Read Security Documentation Hub GitHub

System Overview

Five runtime planes, one narrow responsibility each.

The ReDoor monorepo is service-oriented at runtime. The client and app shell hold the trusted edge. Relay, directory, and evidence services are separate operational planes with distinct security roles.

Topology posture

  • 2 trusted edge planesiOS shell and client runtime keep identity, session state, and wipe logic local.
  • 1 untrusted transport planeRelay storage and delivery stay intentionally opaque.
  • 2 support planesDirectory discovery and evidence commitments remain operationally separate.

Trusted edges, narrow middle

The left side terminates sensitive state. The center only forwards opaque envelopes. Discovery and evidence stay decoupled so no single middle layer becomes a global trust anchor.

Component Responsibilities

Where each service stops.

client/

Client runtime

Maintains identity, prekeys, ratchet sessions, route policy, and diagnostics in Rust. Builds encrypted envelopes and exposes FFI to the mobile shell.

RedoorApp/

iOS application layer

Enforces secure network validation, lifecycle wipe rules, duress flows, and high-risk posture checks before calling into the runtime.

relay-node/

Relay transport plane

Terminates TLS, validates HMAC and replay controls, keeps normal blobs in memory, rotates mailbox handles, and applies abuse governance without seeing plaintext.

directory-dht/

Directory discovery plane

Publishes and resolves signed key material with TTL prekeys, rate limits, optional auth, and lease or sequence ownership semantics.

blockchain-node/

Tamper-evidence plane

Accepts message-hash commitments and signed blocks, persists integrity history on disk, and exposes metrics or admin endpoints with rate and token controls.

End-To-End Flow

Message transport is opaque in the middle and stateful only at the edges.

The sequence matters: resolve signed material, derive sender state, push opaque ciphertext through the relay, then consume the message in device memory before recording commitment evidence.

Flow posture

  • Signed key lookup firstDirectory responses establish the sender’s view of recipient material.
  • Opaque relay secondThe relay sees handles, not usable plaintext.
  • Evidence lastBlockchain writes are commitments, not message content.
1

Identity and prekeys

Device generates identity keys and prekeys and publishes bundle material through the directory plane.

2

Directory resolution

Sender resolves peer key material and verifies signed directory responses before session setup.

3

X3DH and ratchet setup

Session state is initialized in runtime memory using X3DH and Double Ratchet foundations.

4

Envelope construction

Plaintext is serialized as inner payload, ratchet-encrypted, and wrapped into an opaque envelope.

5

Relay forwarding

Relay receives an opaque blob, indexes it by rotating mailbox handle, and serves fetch-once retrieval paths.

6

Evidence submission

Client best-effort submits message-hash commitments to the blockchain plane for integrity evidence.

7

In-memory consume and wipe

Receiver decrypts in memory and lifecycle rules wipe sensitive state on lock, background, terminate, or duress.

Trust Boundaries

Where plaintext exists, and where it should not.

Boundary Trusted for Not trusted for Notes
Device boundary Plaintext, keys, session state, local policy decisions Durable persistence of sensitive runtime material Plaintext should remain in RAM only according to current design intent.
Relay boundary Transport availability, replay checks, abuse throttling Message content confidentiality or social-graph trust Relay sees opaque blobs, timing, and mailbox-level activity.
Directory boundary Publishing and resolving signed key material Blind trust in returned identity data Resolve responses must be signed and verified by clients.
Blockchain boundary Tamper-evidence history and commitment integrity Message secrecy or ciphertext storage Chain stores hashes and linkage metadata, not plaintext messages.

The relay, directory, and blockchain planes are intentionally not treated as places where plaintext should live.

Plaintext Residency

Device memory is the intended plaintext boundary.

Exists

Sender device memory before encryption and receiver device memory after decryption.

Does not exist

Relay queues, directory responses, and blockchain ledger entries.

Architectural Constraints

The current system makes explicit tradeoffs.

  • No global account authority. Identity verification is not framed around a centralized verification service.
  • High-security paths expect onion or mix routing. Strict anonymity mode is treated as a policy boundary, not a marketing extra.
  • Volatile behavior is a product requirement. The mobile shell and runtime are built around wipe and duress flows rather than durable chat history as the primary posture.
  • Integrity evidence is deliberate persistence. The blockchain plane is the one place where durable history is intentional, because it stores commitments rather than message content.
